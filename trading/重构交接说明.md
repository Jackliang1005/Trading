## 当前定位

`trading` 当前定义是统一的 **做T Agent**：

- `trading_core/engine.py`
  负责盘前、盘中检查主循环，串联 selector、decision、execution。
- `trading_core/analysis.py`
  负责行情获取、分时分析、机会评分、信号格式化。
- `trading_core/data_agent.py`
  负责拉持仓股行情、资金流快照。
- `trading_core/news_agent.py`
  负责拉个股、板块、宏观新闻，并抽取事件标签。
- `trading_core/selector_agent.py`
  负责从持仓池里筛出 `focus / watch / avoid`，并产出 AM/PM 模式与做T预算。
- `trading_core/budget_agent.py`
  负责把机动资金预算和持仓卖出预算分配到重点票。
- `trading_core/decision_engine.py`
  负责根据 playbook、行情、模式、预算输出唯一动作。
- `trading_core/risk_engine.py`
  负责自动交易约束、轮数约束、可卖约束。
- `trading_core/review_engine.py`
  负责决策日报、模式复盘。
- `trading_core/storage.py`
  负责配置、当日计划、状态文件的读取与原子写入。
- `trading_core/command_service.py`
  负责飞书指令、初始持仓库维护、当日计划查看、执行回执落地，以及通过 `qmt2http` 直连下单。
- `trading_core/recorder.py`
  负责成交/放弃执行记录与日报统计。
- `trading_core/notifier.py`
  负责飞书发送。
- `trading_core/qmt2http_client.py`
  负责调用 `qmt2http` 的健康检查、下单/撤单接口。

兼容入口仍保留：

- `做T飞书监控.py`
- `做T记录助手.py`
- `record_trade.py`
- `飞书交易Webhook.py`

## 这次重构解决了什么

1. 去掉了监控脚本单文件耦合，后续加策略不会继续堆在一个文件里。
2. `今日交易计划.json`、`monitor_state.json`、交易记录都改为原子写入，降低脏写风险。
3. 飞书 webhook 和监控主程序现在共用同一套 `TradingCommandService`，避免双份逻辑漂移。
4. 飞书提醒升级为“机会评分 + 行动建议”，更接近实际执行决策，而不是纯价格提示。

## 现在的赚钱逻辑

系统的目标不是开新仓，而是在已有持仓池里，先筛出今天最值得做 T 的股票，再把预算、动作和风险单位约束到位：

- 盘前：
  先读取 `初始持仓数据库.json` 作为真实底仓基线，再拉持仓股行情、资金流、个股/板块/宏观新闻。
- 选票：
  自动把持仓池拆成 `focus / watch / avoid`，并给出 `AM/PM 模式`、`买预算 / 卖预算`、建议股数。
- 盘中：
  用 VWAP、分钟均线、分时枢轴、前日支撑压力和分钟 ATR 近似，先识别 `trend_up / trend_down / range / panic_rebound`，再生成 `T买 / T卖` 参考。
- 决策：
  根据 playbook、市场 regime、事件标签、AM/PM 模式、结构、风险单位、预算约束给出唯一动作。
- 通知：
  飞书消息直接告诉你是该“等”“买”“卖”，并附当前模式、预算、建议股数、风险系数、偏好结构和新闻解释。
- 闭环：
  你在飞书里维护初始持仓、查看观察池、下达/执行指令，系统记录当日执行结果、模式复盘，并用近端真实盈亏反调 `risk_factor / entry_tolerance / preferred_structure`。

现在也支持另一种模式：

- 指令执行：
  你可以用飞书直接触发 `qmt2http` 下单，不再只停留在提醒。
- 自动执行：
  仅 `focus` 票允许自动执行，且受 `AM/PM 模式`、`买预算 / 卖预算`、`可卖底仓` 约束。

## 当前关键数据文件

- `初始持仓数据库.json`
  真实底仓、成本价、账户概览的初始真源。系统不再依赖实时持仓查询作为初始化依据。
- `trading_data/news_snapshots_cache.json`
  新闻缓存快照，按盘前/盘中时段刷新。
- `trading_data/hot_concepts_cache.json`
  近 7 天热门题材缓存，盘中分时段刷新当日题材。
- `trading_data/focus_list.json`
  观察池输出，包含 `focus / watch / avoid`、AM/PM 模式、预算、建议股数；当前版本字段为 `stable-v1`。
- `trading_data/universe_state.json`
  持仓池全量筛选结果，包含每只票的 `score_breakdown` 分项得分。
- `trading_data/learning_snapshot.json`
  学习快照，记录每只票的 `样本数 / 胜率 / 均值 / risk_factor / entry_tolerance / preferred_structure`。
- `今日交易计划.json`
  selector 回写后的当日计划，不再只是手工临时覆盖。
- `trading_data/monitor_state.json`
  盘中监控、信号冷却、zone、intraday 状态。
- `trading_data/decision_journal_YYYY-MM-DD.json`
  当日决策流水，供 `决策日报` 和 `模式复盘` 使用。

## 2026-04-01 自动交易链路修复

本轮主要处理了“自动交易不开单”和“提交后状态过于乐观”两个问题。

### 已修复

1. `qmt2http` token 强制统一
   - 所有命中的 `qmt2http` 调用现在默认都会带 `998811`
   - 同时发送 `X-API-Token` 和 `Authorization: Bearer ...`
   - 不再依赖运行时环境变量是否存在

2. 分钟线链路打通
   - `analysis.py` 现在会读取 `做T监控配置.json` 中的 `qmt2http` 配置
   - 分钟线接口空返回时不再直接抛异常
   - 已实测可以正常取到 `300475` 的 1 分钟数据

3. 消息冷却与自动下单解耦
   - 过去自动下单被挂在 `should_send_t_signal(...)` 分支下
   - 这会导致消息冷却命中时，自动交易也一起被挡住
   - 现在提醒仍受冷却控制，但自动交易单独判断

4. 自动交易新增执行过滤
   - 默认要求 `confidence >= 中`
   - `t_spread_pct >= 1.5`
   - 当前执行价距离触发价不超过 `0.5 ATR`
   - 最近 5 根分钟量能均值 / 全日均量 >= `0.6`
   - 阈值已写入 `做T监控配置.json`，可盘中直接调参

5. 未下单原因可直接从日志诊断
   - 现在会显式打印 `decision=wait`、`risk_block=...`、`execution_filter=...`
   - 盘中可以直接区分是信号不足、风控拦截、执行过滤未过，还是交易通道异常

### 执行闭环现状

自动交易状态流转已改为：

`pending/acknowledged -> submitted -> executed`

当前规则：

- 下单请求成功，只会把指令记为 `submitted`
- 监控主循环每轮会调用 `query_today_trades()`，尝试把券商真实成交回报匹配到 `submitted` 指令
- 只有匹配到真实成交后，才会：
  - 更新 `command_book.json` 为 `executed`
  - 更新 `portfolio_state.json`
  - 记入成交日报
  - 回写自动交易计数

这意味着系统不再把“提交成功”误认为“已成交”。

### 当前剩余限制

- `query_today_trades()` 仍属于 OCR / best effort 接口，字段稳定性取决于券商端页面
- 当前匹配逻辑优先识别“完整成交”，对“部分成交、多笔拆分成交”仍偏保守
- 若后续要进一步提高实盘可靠性，下一步应继续补“部分成交闭环”和“盘口/滑点过滤”
