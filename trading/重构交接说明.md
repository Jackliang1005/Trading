# Trading 重构交接说明

## 当前定位

`trading` 现在不再只是“价格碰区间提醒脚本”，而是统一的 **做T Agent**：

- `trading_core/engine.py`
  负责盘前、盘中检查主循环，串联 selector、decision、execution。
- `trading_core/analysis.py`
  负责行情获取、分时分析、机会评分、信号格式化。
- `trading_core/data_agent.py`
  负责拉持仓股行情、资金流快照。
- `trading_core/news_agent.py`
  负责拉个股、板块、宏观新闻，并抽取事件标签。
- `trading_core/selector_agent.py`
  负责从持仓池里筛出 `focus / watch / avoid`，并产出 AM/PM 模式与做T预算。
- `trading_core/budget_agent.py`
  负责把机动资金预算和持仓卖出预算分配到重点票。
- `trading_core/decision_engine.py`
  负责根据 playbook、行情、模式、预算输出唯一动作。
- `trading_core/risk_engine.py`
  负责自动交易约束、轮数约束、可卖约束。
- `trading_core/review_engine.py`
  负责决策日报、模式复盘。
- `trading_core/storage.py`
  负责配置、当日计划、状态文件的读取与原子写入。
- `trading_core/command_service.py`
  负责飞书指令、初始持仓库维护、当日计划查看、执行回执落地，以及通过 `qmt2http` 直连下单。
- `trading_core/recorder.py`
  负责成交/放弃执行记录与日报统计。
- `trading_core/notifier.py`
  负责飞书发送。
- `trading_core/qmt2http_client.py`
  负责调用 `qmt2http` 的健康检查、下单/撤单接口。

兼容入口仍保留：

- `做T飞书监控.py`
- `做T记录助手.py`
- `record_trade.py`
- `飞书交易Webhook.py`

## 这次重构解决了什么

1. 去掉了监控脚本单文件耦合，后续加策略不会继续堆在一个文件里。
2. `今日交易计划.json`、`monitor_state.json`、交易记录都改为原子写入，降低脏写风险。
3. 飞书 webhook 和监控主程序现在共用同一套 `TradingCommandService`，避免双份逻辑漂移。
4. 飞书提醒升级为“机会评分 + 行动建议”，更接近实际执行决策，而不是纯价格提示。

## 现在的赚钱逻辑

系统的目标不是开新仓，而是在已有持仓池里，先筛出今天最值得做 T 的股票，再把预算和动作约束到位：

- 盘前：
  先读取 `初始持仓数据库.json` 作为真实底仓基线，再拉持仓股行情、资金流、个股/板块/宏观新闻。
- 选票：
  自动把持仓池拆成 `focus / watch / avoid`，并给出 `AM/PM 模式`、`买预算 / 卖预算`、建议股数。
- 盘中：
  用 VWAP、分钟均线、分时枢轴、前日支撑压力，给出 `T买 / T卖` 参考。
- 决策：
  根据 playbook、市场 regime、事件标签、AM/PM 模式、预算约束给出唯一动作。
- 通知：
  飞书消息直接告诉你是该“等”“买”“卖”，并附当前模式、预算、建议股数和新闻解释。
- 闭环：
  你在飞书里维护初始持仓、查看观察池、下达/执行指令，系统记录当日执行结果和模式复盘。

现在也支持另一种模式：

- 指令执行：
  你可以用飞书直接触发 `qmt2http` 下单，不再只停留在提醒。
- 自动执行：
  仅 `focus` 票允许自动执行，且受 `AM/PM 模式`、`买预算 / 卖预算`、`可卖底仓` 约束。

## 当前关键数据文件

- `初始持仓数据库.json`
  真实底仓、成本价、账户概览的初始真源。系统不再依赖实时持仓查询作为初始化依据。
- `trading_data/news_snapshots_cache.json`
  新闻缓存快照，按盘前/盘中时段刷新。
- `trading_data/hot_concepts_cache.json`
  近 7 天热门题材缓存，盘中分时段刷新当日题材。
- `trading_data/focus_list.json`
  盘前观察池输出，包含 `focus / watch / avoid`、AM/PM 模式、预算、建议股数。
- `trading_data/universe_state.json`
  持仓池全量筛选结果。
- `今日交易计划.json`
  selector 回写后的当日计划，不再只是手工临时覆盖。
- `trading_data/monitor_state.json`
  盘中监控、信号冷却、zone、intraday 状态。
- `trading_data/decision_journal_YYYY-MM-DD.json`
  当日决策流水，供 `决策日报` 和 `模式复盘` 使用。

## 2026-03-28 新增通道降级与上下文刷新机制

本轮补充了两层运行机制：

1. `qmt2http` 通道降级
   - 系统只用 `health.trade_connected` 判断买卖通道是否可用
   - 不再依赖 `positions` 等账户查询接口判断是否可交易
   - 若交易接口断开，则自动降级为“只监控，不自动交易”
   - 持仓和仓位状态继续以 `初始持仓数据库.json` + `portfolio_state.json` 为准

2. 新闻/题材时段刷新
   - 新闻按 `preopen/open/mid_am/late_am/early_pm/late_pm` 分时段刷新
   - 热门题材维护近 7 天缓存，并在盘中关键时段刷新当日题材
   - 盘中进入新时段时，系统会重算 `focus / watch / avoid`

飞书新增查看命令：

- `T 信息状态`
  查看新闻缓存、题材缓存的更新时间与当前刷新规则
- `T 通道状态`
  查看 qmt2http 当前是否可达、行情是否正常、交易是否正常

建议使用顺序：

1. 先用 `T 直买 / T 直卖` 验证链路
2. 再用 `T 执行 任务ID` 验证派单模式
3. 最后再考虑打开 `auto_trade_enabled`
