## 一、项目目标

这套项目的目标不是做一个“价格碰区间就提醒”的简单脚本，而是一个 **持仓型半自动做T Agent**：

1. 围绕真实持仓池做盘前筛选、盘中决策、执行约束和复盘。
2. 综合行情、分时结构、资金流、新闻、题材、外盘同行风险，生成 `focus / watch / avoid` 和 `AM/PM mode`。
3. 通过飞书查看观察池、今日计划、仓位状态、指令簿，并可直接更新当日计划或下单。
4. 飞书入口采用 **webhook 事件驱动**，监控主循环与交易指令解耦。

## 三、当前默认逻辑

当前不是简单“默认启用哪几只股票”，而是：

1. 先用 [初始持仓数据库.json](/root/.openclaw/workspace/trading/初始持仓数据库.json) 作为真实持仓基线
2. 盘前拉持仓股行情、资金流、个股/板块/宏观新闻
3. 生成 `focus / watch / avoid`
4. 盘中按分钟线识别 `trend_up / trend_down / range / panic_rebound / neutral`
5. 用波动单位生成 `T买 / T卖`
6. 给每只重点票生成 `AM/PM mode`、`买预算 / 卖预算`、建议股数
7. 再用近端真实盈亏回算 `risk_factor / entry_tolerance / preferred_structure`
8. selector 回写 [今日交易计划.json](/root/.openclaw/workspace/trading/今日交易计划.json)

当前真实持仓以 [初始持仓数据库.json](/root/.openclaw/workspace/trading/初始持仓数据库.json) 为准，而不是手工记忆。
当前 `qmt2http` 只承担两类职责：
- 行情/分钟线/健康检查
- 买入 / 卖出 / 撤单
- 持仓查询 / 账户资产查询 / 今日委托查询 / 今日成交查询（best effort，OCR 依赖）

不承担：
- 持仓初始化（真源为 `初始持仓数据库.json`）
- 做T预算基线

做T监控脚本（做T飞书监控.py）原先有两个已知 gap：
1. 价格落入区间就反复报警，靠 20 分钟冷却抑制噪音，但不够精确
2. 触发止损时没有任何上下文信息，用户收到提醒后需要自己去查原因

注意：qmt2http 的持仓/账户接口不作为本系统持仓真源，不接入持仓初始化和预算基线。

### 改动 1：首次穿越边缘检测

新增函数：`classify_zone`、`is_first_crossing`、`fetch_minute_prices`

逻辑：
```
state.zones 记录每只股票上一轮所在区域
├─ zone 变化 (如 between → buy) → 一定是首次穿越 → 发信号
├─ zone 未变 → 查 qmt2http 最近 5 根分钟线
│   ├─ 分钟线中有"先离开区间再回来" → 允许发信号
│   └─ 分钟线一直在区间内 → 抑制（不是新穿越）
└─ qmt2http 不可用（夜间/离线） → zone 未变就抑制（保守降噪）
```

区域划分：`risk` / `below_buy` / `buy` / `between` / `sell` / `above_sell`

降级策略：qmt2http 超时设为 10 秒，不可用时自动退化为 zone 变化判断，不影响主流程。

### 改动 1.1：青云科技改为“破位后反弹观察”

背景：
- 2026-03-24 盘中，`688316 青云科技-U` 多次跌破原风险线 `74.8`
- 旧逻辑里，`price <= stop_loss` 会直接判定为 `risk`
- 结果是这类高波动、轻仓逆T票一旦破位，全天只会收到“风险提醒”，不会再出现反弹观察信号

本次调整：
- 仅对开启配置的个股生效，不影响其他默认标的
- 新增配置项：
  - `allow_rebound_watch_after_stop`
  - `rebound_buy_above`
- 含义：
  - 跌破 `stop_loss` 后，先进入防守，不再按原买入区间直接给普通低吸提醒
  - 若后续重新站回 `rebound_buy_above` 上方，允许触发一次“反弹观察提醒”
  - 该提醒明确限定为：
    - 仅适用于已有先卖后买仓位
    - 仅适合分时承接明确时的小仓回补
    - 不等于抄底摊平

当前青云科技配置：
- `stop_loss = 74.8`
- `allow_rebound_watch_after_stop = true`
- `rebound_buy_above = 75.2`

策略意图：
- `74.8` 现在更接近“原计划失效，切换防守”的阈值
- 不是“全天彻底禁买”的最终否决线
- 对“轻仓逆T”这类票，更合理的动作是：
  - 先承认破位
  - 再等反弹确认
  - 最后只给观察性回补提醒

### 改动 1.2：美利云改为“轻观察模式”

背景：
- `000815 美利云` 原先为 `enabled = false`

## 十二、2026-03-21 更新：webhook 签名校验 + event_id 去重

### 背景

飞书交易 webhook（`feishu_trading_webhook.py`）原先有两个安全/可靠性 gap：
1. 无签名校验：任何人 POST 到 `/feishu/trading` 都会被处理
2. 无事件去重：飞书未收到 200 时会重试，可能导致同一指令被执行多次

### 改动文件

`/root/.openclaw/workspace/skills/feishu-trading-webhook/scripts/feishu_trading_webhook.py`

### 改动 1：签名校验

飞书事件订阅 v2 签名算法：`SHA256(timestamp + nonce + encrypt_key + body)`

处理流程：
```
请求到达
├─ 读取 header: X-Lark-Request-Timestamp, X-Lark-Request-Nonce, X-Lark-Signature
├─ 用 FEISHU_ENCRYPT_KEY + 上述参数计算签名
├─ hmac.compare_digest 比对（防时序攻击）
├─ 不匹配 → 403 + 写日志
└─ 匹配 → 继续处理
```

同时对 challenge 请求也校验 `verification_token`。

环境变量：
- `FEISHU_ENCRYPT_KEY`：飞书开放平台 > 事件订阅 > Encrypt Key
- `FEISHU_VERIFICATION_TOKEN`：飞书开放平台 > 事件订阅 > Verification Token

向后兼容：未配置环境变量时跳过校验，行为与之前一致。

### 改动 2：event_id 幂等去重

飞书每次推送事件时，`header.event_id` 是唯一标识。相同 event_id 的重试推送应跳过。

实现：
- 内存 dict `_SEEN_EVENTS`：`{event_id: timestamp}`
- 首次见到 → 记录并处理
- 再次见到 → 立即返回 `{"ok": true, "duplicate": true}`，不执行指令
- 容量上限 1000 条，超过时淘汰 10 分钟前的旧条目
- 空 event_id 不去重（兼容异常情况）

### 改动 3：日志增强

日志中新增 `event_id` 字段，方便排查重复推送。
签名校验失败、token 不匹配、重复事件均写入日志。

### 改动 4：启动信息

启动时打印安全配置状态：
```
签名校验: 已启用 / 未配置 (FEISHU_ENCRYPT_KEY)
Token校验: 已启用 / 未配置 (FEISHU_VERIFICATION_TOKEN)
事件去重: 已启用 (TTL=600s, max=1000)
listening on http://0.0.0.0:8788/feishu/trading
```

### 配置方式

```bash
# 在 .bashrc 或启动脚本中添加（从飞书开放平台获取）
export FEISHU_ENCRYPT_KEY="你的encrypt_key"
export FEISHU_VERIFICATION_TOKEN="你的verification_token"
```

2026-03-21 单元测试全部通过：
- 签名计算与手动 SHA256 结果一致
- hmac.compare_digest 比对正确
- 首次 event_id 不去重
- 重复 event_id 正确去重
- 不同 event_id 互不影响
- 空 event_id 永不去重
- 超过 TTL 的旧事件被正确淘汰

## 十三、2026-03-26 更新：基于分时数据的精确做T信号

### 背景

做T监控系统原先只能在价格进入静态配置的 buy_range/sell_range 时提醒"可以做顺T/逆T"，无法给出精确的做T价位建议。更严重的问题是，像香农芯创这样的票，现价约138，配置区间154-156，价格完全脱离了配置区间，系统只能反复报"risk"，毫无参考价值。

### 新增功能

#### 1. 分时分析引擎

新增 `IntradayAnalysis` 数据结构和 `compute_intraday_analysis` 核心分析函数：

- **VWAP 计算**: `sum(typical_price * volume) / sum(volume)`
- **枢轴点检测**: 滑窗法（window=3）识别分时K线的局部高/低点
- **前日枢轴**: PP=(H+L+C)/3, S1=2*PP-H, R1=2*PP-L
- **分钟均线**: MA20, MA60
- **综合目标价**:
  - T买入 = 加权平均(最近支撑枢轴 30%, VWAP下轨 25%, 前日S1 25%, MA支撑 20%)
  - T卖出 = 加权平均(最近压力枢轴 30%, VWAP上轨 25%, 前日R1 25%, MA压力 20%)
- **置信度评估**: 多个指标在 0.5% 范围内聚集程度 → 高/中/低

#### 2. 策略差异化调整 (`adjust_for_strategy`)

- **顺T优先**: T-buy 需回落到 VWAP 下方
- **轻仓逆T**: 置信度降一级
- **箱体震荡**: 价差过小时降低置信度
- **趋势观察(light)**: 仅生成 T-sell 信号

#### 3. 动态区间替代

当现价与配置的 buy_range 中点偏离超过 8% 时，自动切换为动态区间：

- 动态买入区: `[t_buy_target * 0.995, t_buy_target * 1.005]`
- 动态卖出区: `[t_sell_target * 0.995, t_sell_target * 1.005]`
- 动态止损: 当日最低枢轴点下方 1%
- 飞书消息中标注"[动态区间]"

#### 4. 独立 T 信号

当无 zone 信号触发但分析有高/中置信度 T 机会时，发独立 T 信号消息：

- 独立冷却：每只票 30 分钟间隔
- 每日上限：每只票最多 4 条
- 仅在 steady/winddown 阶段且现价接近 T 买入或 T 卖出参考价时触发

#### 5. 数据源

| 数据接口 | 用途 | 缓存策略 |
|---|---|---|
| `get_intraday_minute_data` (全天) | 完整1分钟OHLCV | 每轮获取 |
| `get_history_data` (period=1d, count=2) | 前日日K线 | 每日首次获取后缓存 |

### 时段划分

| 阶段 | 时段 | 行为 |
|---|---|---|
| bootstrap | 09:25-09:45 | 不做分时分析 |
| first_wave | 09:45-10:00 | 开始分析（K线数可能不够） |
| steady | 10:00-14:30 | 稳态分析阶段 |
| winddown | 14:30-15:00 | 尾盘收敛 |
| closed | 其他 | 不分析 |
